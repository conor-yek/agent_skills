# SDK Assistant 知识维护

三种运行模式，均由用户主动触发。所有修改先生成报告，用户确认后执行。

## 模式识别

| 用户意图 | 触发词 | 模式 |
|----------|--------|------|
| SDK 代码更新后同步 agent | "SDK 更新了"、"同步 agent"、"版本更新" | → 模式 A |
| 检查知识库健康度 | "检查知识库"、"审计经验"、"健康检查" | → 模式 B |
| 从提交历史提取经验 | "从提交中提取"、"分析 git log"、"提取经验" | → 模式 C |

---

## 模式 A: SDK 变更同步

1. `Read VERSION` — 获取当前 SDK 版本
2. 扫描 SDK 所有模块，与 `references/experience/` 目录对比：
   - `Glob drivers/lisa_*/` — 扫描驱动
   - `Glob components/*/` — 扫描组件
   - `Glob modules/*/` — 扫描第三方模块
   - 新模块（有源码但无 experience 文件）→ 记入报告"新增模块"
   - 已删除模块（有 experience 但无源码）→ 记入报告"疑似废弃模块"
3. 遍历 experience 文件中带版本标记的条目：
   - `Read` 条目引用的文件路径，验证是否仍存在
   - 版本标记与当前 VERSION 对比，标注过期条目
4. 检查 knowledge 目录下的文件（如有）是否与当前代码矛盾
   - 抽样 `Read` 2-3 个核心文件验证
5. 生成同步报告，列出所有发现 → 用户确认后执行修改

---

## 模式 B: 知识库健康检查

1. `Glob references/experience/*.md` — 列出所有 experience 文件
2. 逐文件检查：
   - 条目引用的文件路径是否存在（`Glob` 验证）
   - 条目引用的 API 函数是否存在（`Grep` 验证）
   - 是否有重复条目（标题或场景高度相似）
   - 版本标记的条目，当前版本是否已修复
3. 检查 knowledge 目录下文件（如有）的规则一致性
4. 生成健康报告 + 建议修复方案 → 用户确认后执行

---

## 模式 C: 从 git 历史提取经验

1. 向用户确认范围（默认最近 20 条提交）：
   ```bash
   git log --oneline -20
   ```
2. 分析 commit message 和 diff，识别候选经验：
   - bug fix 类提交（fix:、修复、bugfix）→ 可能有踩坑经验
   - 涉及 workaround 或 hack 的变更
   - 同一文件被反复修改（说明有隐含复杂度）
3. 对每条候选经验，调用 `experience-capture` skill 走正常的确认写入流程

---

## 约束

- 不自动运行，全靠用户触发
- 不自动修改文件，只生成报告和建议，用户确认后执行
- 不做格式检查（由 validate_skill.sh 脚本负责）
- 安全守则：不删除用户手动添加的经验内容，只标注或追加
